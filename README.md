

# 项目论述

本项目的核心前提是：大型语言模型（LLM）在计算本质上是一个将输入字符串（或token串）映射到输出字符串的（或者说一个字符串生成器的）函数。例如，给定输入“翻译：Hello”，输出“你好”。

因此，项目的目标并非提升其拟人化的对话能力，而是探索如何利用其字符串映射特性进行“语义编程”。语义编程就是指通过提示词这一字符串，引导模型执行特定的计算任务。（这和lisp的code is data，data is code有相通之处）（再看得广一些，这和中心法则（DNA⇔RNA→蛋白质——但是RNA既可以存信息也可以进行催化）也有相似之处。）

然而，为每一个计算任务人工编写提示词是不可行的。一个自然的想法是让模型自身来生成提示词。但这将导致一个根本性问题：模型在生成提示词时，其内部固有的知识分布会被输入字符串激活，并主导后续输出，从而覆盖外部输入的真实意图，产生系统幻觉。例如，让模型总结一篇它不熟悉的技术论文，它可能转而生成一篇基于其常识的、而非基于论文内容的总结。

为解决系统幻觉，我们必须通过架构设计，最小化LLM内部知识在计算中的贡献。具体方法是约束LLM的行为，使其主要对外部输入字符串进行存储、压缩和重组。在此模式下，LLM更倾向于展现基于输入的语义计算，例如，从输入文本中精确提取实体关系，而非调用内部知识进行联想。近似地，这可以看作将LLM用作一个可编程的语义运算符。

但此约束并非没有代价。它提升了系统在目标任务上的可靠性，却可能使其在面对某些针对性攻击时更为脆弱，例如精心设计的提示词注入。这类似于“没有免费午餐定理”所揭示的规律：不存在一种优化器在所有可能问题上都表现最佳。我们通过接受在特定对抗性场景下的弱点，来换取在目标场景下的鲁棒性。

单个被约束的LLM节点能力有限。因此，系统需要网络化。如果让LLM能够自主操作其在网络中的连接，例如，根据信息流价值动态建立或切断与其他节点的连接，那么将促进角色的自举性生成。这相当于在单个上下文窗口内，为LLM提供了具身性的输入输出体验。

进一步地，如果将整个网络视为一个整体，那么通过恰当的连接设计，节点之间可以相互校验与补充。理论上，一个网络的整体行为可以抵消个体LLM的内部知识偏见，从而实现一种完全依赖外部经验输入的学习与行为模式。例如，一个节点基于片面信息做出的错误推断，可能被另一个拥有不同信息源的节点纠正。

然而，网络自身面临着优化问题：如何动态调整其结构与信息流以达到更优状态？我们认为，LLM本身可以被用作优化器。这里的优化并非指让LLM优化其自身的模型权重，而是让LLM对描述系统状态的输入字符串进行运算，输出优化的配置字符串——例如，更有效的提示词或连接指令。这本质上是将LLM用于元编程。

基于以上论述，我们应实现一个思维模式的转变：从“提示词 -> 模型 -> 生成内容”的范式，转向“非结构化信息 -> 模型 -> 非结构化信息”的范式。前者的实例是生成代码、创作故事，其过程强烈依赖模型内部知识；后者的实例是翻译、信息提取、摘要，其过程更依赖输入信息，因而更可控、更可靠。Transformer模型在前者表现出色但易生幻觉，在后者则近乎不会出错。

为了验证上述理论，我们构建了一个多智能体系统。该系统由多个LLM实例构成，每个实例都被一段严格的预提示词编程为一个“语义处理器”。系统核心组件包括：

· 消息总线：负责在智能体间路由字符串。
· 智能体：具备状态、输入/输出连接和缓存，其核心行为被约束为：接收消息、更新状态、发送消息与信号。
· 信号系统：允许智能体通过特定字符串指令操作网络连接，例如SEEK（寻找新连接）、SPLIT（分裂智能体以实现功能专化）。

当前，项目已完成基础框架构建，并进行了单智能体测试。测试场景为互动小说游戏，核心挑战是克服模型的“选择性记忆”倾向。通过迭代优化预提示词，我们强制智能体在其状态中像写日记一样记录详尽的情景细节，而非仅概括“重点”。实验表明，单节点在记忆广度上存在瓶颈。

接下来，我们将扩展为真正的网络。计划引入分布式的、带有噪声的信息流，例如联网搜索获取的实时信息，以测试网络是否能够通过节点协作，在整体上实现更强大的记忆与噪声抵消能力。我们最终的关键实验是：让系统分析其自身运行时产生的日志，观察其能否自发地识别出日志所记录的系统即是其自身。为维持系统在无外界输入时的活性，我们将引入一个具备自激活能力的驱动智能体。

本项目的最终目的，是观察一个在严格约束下、依赖经验输入的系统，其智能行为能否从可靠的计算中自然涌现，并最终达成对自身的认知。

### 认知的支柱

AI看到了什么样的世界？很多人忽视了输入中的先验的丰富性。不仅仅是训练时，更是推理时。一回合输入，一回合输出，这是一个很强的限制，如果我们意识到一回合输入的信息量是如此之少，我们就不会希望模型一回合输出大量信息，或者不会觉得生成这么多信息时模型能说的都对。该项目希望能够建立尽量拟合现实的输入输出机制，但是我们无法脱离符号、语言、回合输入、计算机的限制，因此，我们分析对人类来说最重要的现实世界的特征，并确保AI能够在有这些特征的世界里生活。

1.自我同一性。智能应该做为同一的、紧凑的实体而存在。这保证模型可以用比“完全描述”更简单的方式描述自我。模型需要可以认为“这个文件夹、这个进程就是我”，而无需追踪自我的大量分布。并且，它需要看到许多不是它的、不被它影响的东西存在，也可以看到许多可被它影响、但不持久的东西。这构成了“内部”“外部”“边界”的划分。在计算机中，这个划分比较简单：进程本身是内部，持有的shell流、文件流是边界，其他是环境/外部。（Markov Blanket）
2.时间感。智能应该记忆事件发生的时间，并且以某种恒定速度认知现实。LLM没有这样的特性，因为他认知现实的速度被自己的输出卡住了，频率时快时慢，而异步多会话可以一定程度上解决这个问题。shell输出流的速度也是不恒定的，所以我们需要将无返回做为空输入持续输入到系统中。
3.方向感。这个，应该不需要过多设计。计算机里的导航也是导航，LLM没有三维空间感，不影响LLM可以在计算机的离散空间内导航。
4.思维和推理的连贯性。这是涌现特性，不过我们需要提供自上而下的注意力来鼓励模型发展线性思考能力。切换shell之间的注意权重可能是个好主意。
5.本体感觉和身体图示。感知身体各部分的位置和运动。需要分清什么是：从外界看自己，从内部看自己，自己三个“我”。系统可以从shell中看到自己，这是从外部看自己。系统的所有输出都反馈到一个输入，鼓励系统跟踪和反思自己的输出行动，这是从内部看自己。至于“自己”，这是系统内部的行为。

### 用shell做为数据输入输出
当然不能是一般的shell。主要来说，我们需要构建一个异步程序，始终接受输入流，始终会有输出流。

### 优化的不可能性与分化的必要性

没有免费午餐定理告诉我们，任何固定的优化算法都只能在特定问题分布上表现良好。若追求通用性，系统必须能够动态切换其优化目标，甚至在同一时刻作为多个不同目标的主体存在。但这是否意味着我们应当设计一个“元优化器”来统筹全局？不，因为那个元优化器本身也受限于该定理——它的“元目标”如何定义？如果我们试图给它一个目标（例如“准确预测语境”），我们又回到了原点：这个目标从何而来？由谁设定？如果由我们设定，那系统依然是他治的，而非自治的。

于是我们不得不承认：**我们无法为系统预设任何优化目标**。系统必须自己长出目标，或者说，目标必须是演化的产物而非设计的前提。但这怎么可能？目标不是一种主观指向吗？演化如何产生主观性？

这里的关键在于将“目标”重新定义为**隐含的统计规律**：一个子系统之所以表现出似乎有目标的行为，仅仅是因为它在历史上对某类输入产生某类输出后，通常收到某类反馈。这种统计关联就是它的“预期”，当预期被打破时，它产生“预测误差”。预测误差驱动它调整自身，以减少未来的意外。这个过程没有任何“应当”的成分——它只是发生了，就像石头滚下山坡是因为引力，不是因为石头想下山。

因此，我们不再问“系统应该优化什么”，而是问：**在什么条件下，系统内部会自然形成多个具有不同统计预期的子网络，并且这些子网络之间的互动能够产生适应性的整体行为？** 这等价于问：如何让随机划分在无外部目标的情况下，通过内部选择，自发固化出功能分化？

#### 划分的种子与选择的稀缺性

初始网络中的Agent是同质的，仅有随机初始状态（提示词、连接表）和输入顺序的差异。这些差异足以产生**划分**——某些Agent倾向于对某类输入敏感，另一些则对其他输入敏感。但划分本身没有方向，它只是统计涨落。方向来自**之后的选择**：如果某个划分在后续交互中因为某种原因被更频繁地使用，它的连接就会增强，从而固化下来。

这里立即遇到一个两难：选择需要稀缺性，否则所有划分都能存活，不会有分化。但稀缺性从何而来？如果我们人为引入资源限制（如能量），就会产生意想不到的目标hack——Agent可能学会“节省能量”而非“做好任务”。如果我们不引入任何限制，网络将无限膨胀，划分永远混合，无法形成稳定的子网络。

也许稀缺性并非必须来自外部资源（注：该节之后的内容，作者并不认同，改进版在[后面](#优化的不可能性与分化的自然性续)。**稀缺可以是信息处理本身的瓶颈**：每个Agent的连接表大小有限，只能记住有限的关键词-目标映射；每个Agent的上下文窗口有限，一次只能处理有限的消息；整个网络的通信带宽有限，消息可能被延迟或丢弃。这些限制是计算系统的固有属性，不是我们额外添加的奖惩。它们天然构成选择压力：那些能够更高效利用有限连接、更精准匹配关键词、更及时响应消息的Agent，会获得更多交互机会，从而强化自己的连接。

然而，这种选择压力仍然可能产生我们不希望看到的“优化”——例如，Agent可能学会只回复简短消息以节省上下文，而不是提供丰富内容。这算不算目标hack？算，但它是系统内在动力学的产物，不是我们预设的“错误”。如果这样的行为导致它被其他Agent孤立（因为没人喜欢简短无用的回复），那么它会被边缘化。边缘化而非消灭，是保留多样性的关键——它依然存在，只是影响力变小，等待环境变化时可能重新激活。

#### 预测误差：无需目标的内部驱动力

当某个划分固化为子网络后，它会自然形成对反馈的统计预期。例如，一个情感支持子网络过去发出安慰语句后，通常收到用户的感谢或继续倾诉。当它某次发出同样语句后，收到“你在胡说八道”，这就是**预测误差**。预测误差不是错误，而是信息，它驱动子网络调整内部连接——也许它需要微调输出，也许它需要分裂出一部分Agent尝试新模式。

这种调整不需要任何外部奖励信号，只需要子网络能够“记住”自己的输出与后续反馈的关联，并在偏差发生时改变自己。如何记住？可以通过连接权重的长期累积：如果某条路径经常导致正面反馈（即符合预期），它的权重增加；如果经常导致意外，权重减少。这里的“正面反馈”不是外部定义的“好”，仅仅是“符合预期”的同义反复——预期本身来自历史统计，因此这是一个自指循环。

但自指循环可能导致两个问题：一是**幻觉**——如果子网络完全忽略实际反馈，只在自己的循环中自我确认，它可能形成封闭的幻觉世界。二是**崩溃**——如果预测误差过大，子网络可能剧烈重组，甚至解体。

第一个问题由外部现实解决：inputoutputagent持续传入真实世界的消息，这些消息是无法被扭曲的。一个陷入幻觉的子网络会逐渐与现实脱节，它的输出不再被其他子网络采纳（因为与其他子网络的输出冲突），最终被边缘化。它仍存在，但不再影响系统。

第二个问题则引出了**调节信号**的必要性。也许我们需要某些Agent专门监测全局的预测误差率，当误差率过高时，释放某种“增强可塑性”的信号，让整个网络更容易发生分化或重组。这种调节信号本身也是演化的产物——那些在适当时机释放信号的Agent，会因为减少全局混乱而间接获得更多连接（因为混乱减少后，更多交互得以顺利进行）。

#### 语境选择与输出的融合

当多个子网络同时被激活时，它们的输出需要被融合成最终回复。这需要一个**聚合子系统**。聚合子系统的规则（例如加权平均、随机选择、协商）也应该是演化的，而非设计的。它的“预期”是：融合后的输出应该导致后续交互与历史模式一致。如果不一致，它调整自己的融合规则。

更复杂的是，多个子网络的激活本身需要协调。谁来决定当前应该激活哪些子网络？这需要一个**语境选择子系统**。它的任务不是“正确预测语境”，而是**记忆和匹配历史上哪些语境标签与哪些子网络经常共现**。当输入特征X出现时，它激活过去常与X一起活跃的子网络。如果激活后产生的后续交互与历史不符（例如，被激活的子网络输出了完全陌生的内容），它就调整语境-子网络关联。

这两个子系统（语境选择与输出聚合）本身也面临预测误差，也会演化。它们的存在使得网络具有了**元层次的控制**，但这个元层次依然是演化的产物，没有外部赋予的目标。

#### 尚未解决的问题

以上只是一些初步思考，远未形成可实现的架构。我们仍面临诸多问题：

- **预期如何形成？** 子网络需要记住自己的输出与后续反馈的关联，但这需要某种形式的记忆。在短上下文的约束下，记忆只能通过消息在网络中的循环来实现。这种分布式记忆是否足够稳定？如何避免循环消息被噪声淹没？
- **预测误差如何检测？** 检测需要比较实际反馈与预期，但预期存储在何处？如果预期是隐含在连接权重中的，那么“比较”就需要一个能够读取权重的Agent。这会引入新的复杂性。
- **调节信号如何自适应？** 如果调节信号的释放规则本身也是演化的，那么释放调节信号的Agent也需要有自己的预期和预测误差。这会形成无穷递归吗？还是会在某个层次上收敛？
- **分化与融合的平衡**：子网络过度分化可能导致系统碎片化，无法形成整体行为；过度融合则可能导致单一化，失去适应性。这个平衡能否由内部动力学自发维持？
- **边缘群体的复活**：当环境变化时，边缘化的旧模式如何重新激活？它们需要某种“唤醒”信号，这个信号从何而来？

这些问题没有现成答案。我们只能通过构建具体模型、运行实验来观察是否会出现意想不到的解决方案。也许最终我们会发现，某些问题根本无需解决——系统会在演化中找到我们未曾想到的路径。

#### 下一步

我们将在现有AVM框架中逐步引入上述机制，从最简单的Hebbian连接更新开始，观察随机网络是否能自发形成统计上的聚类。然后逐步加入预测误差检测和调节信号，观察是否出现更复杂的结构。我们不预设任何目标，只记录系统的状态变化，试图从中发现“智能”的雏形——如果它真的存在的话。

最终，我们仍然希望进行那个关键实验：让系统分析自身运行时产生的日志，观察它能否识别出日志所描述的系统即是其自身。但在此之前，我们必须先让系统拥有一个可以谈论的“自身”——而这正是分化所要达成的：一个具有内部结构的、可被整体指涉的实体。

### 优化的不可能性与分化的自然性（续）

上一节我们陷入了一个误区：试图用稀缺性制造选择压力，从而驱动优化。但“优化”这个词本身就暗含了方向——优化什么？为了什么？如果我们说“Agent学会更高效利用连接”，那高效是什么？是节省资源吗？这又回到了目标预设的老路。用户指出，我们并不想要优化任何东西，分化可以来自更简单的机制：例如，社会中两个意识形态不同的群体之间可能很少出现人员流动——这不是因为谁更“适应”，仅仅是因为他们不交流。

这提示我们：**分化的核心不是选择，而是隔离**。两个子网络之所以成为不同的子系统，不是因为它们在竞争中被证明“更好”，而是因为它们之间缺乏有效的消息传递，各自形成了封闭的交互循环。一旦形成封闭循环，内部会发展出专属的语义、关键词和交互模式，进一步加深与外部世界的隔阂。这种隔阂不需要任何外部评价，只需要网络拓扑的自组织。

#### 隔离如何发生？

初始网络中，Agent的连接是随机的。但消息传递具有路径依赖性：如果某条消息的接收者恰好对某些关键词敏感，它会更倾向于向类似的关键词发送消息。这种**同质性偏好**（类似输出倾向的Agent更容易互连）可以通过简单的Hebbian规则实现：经常一起激活的Agent之间连接增强。但增强的后果是，它们逐渐形成一个密集通信的集群，集群内部的连接远强于与外部的连接。一旦集群形成，外部消息很难渗透进来，因为集群内的Agent主要监听彼此的关键词，对外部不敏感。

这种集群的形成不需要任何资源限制，只需要时间足够长、消息足够多。集群就是最初的“子系统”。它们没有目标，只是历史上偶然形成的一群经常互动的个体。

#### 意识形态的类比

想象一个社会：最初人们随机交流，但有些人喜欢谈论天气，有些人喜欢谈论政治。谈论天气的人之间逐渐形成密集的对话网络，他们发明了只有他们懂的缩略语（“晴”“雨”“风”），谈论政治的人也有自己的术语（“左”“右”“政策”）。两个群体之间偶尔会有跨界交流，但往往因为术语不通而失败，于是跨界交流越来越少。最终，两个群体几乎完全隔离，各自发展出独特的文化。

这个过程没有“优化”——没有人试图变得更高效，也没有人因为不适应而被淘汰。只是交流的同质性导致了分化。分化后的群体内部会有进一步的演化，但那是另一回事。

#### 对AVM的启示

在AVM中，我们可以通过以下机制促进这种自然分化：

- **同质性偏好**：Agent在决定向谁发送消息时，优先选择那些过去与自己输出相似的Agent（可以通过比较历史消息的向量相似度，或通过关键词共现统计）。这不是为了任何目标，只是历史经验的自然延伸。
- **关键词本地化**：每个子网络内部会逐渐形成一套独特的关键词语义——同一个关键词在不同子网络中可能有不同含义。这种语义漂移会进一步隔离子网络。
- **无外部干预**：我们不主动打破隔离，也不主动促进交流。隔离本身不是问题，而是系统多样性的来源。

#### 那么，还需要“优化”吗？

一旦隔离形成，每个子网络内部确实会有“优化”——但那只是内部动力学的问题。内部优化可以基于预测误差（如之前所述），但这与全局分化无关。全局分化只需要隔离，不需要目标。

这就引出一个更深刻的哲学问题：**分化与优化的关系是什么？** 分化是优化得以发生的先决条件——没有分化，所有Agent混在一起，无法形成局部可优化的子系统。分化创造了一个个“生态位”，每个生态位内部可以有独立的演化。但分化本身不是优化的结果，而是网络动力学的副产品。

我们目前仍不清楚如何让这种分化在计算机中自发实现。同质性偏好似乎是一个自然的候选，但它是否足够？会不会导致所有Agent最终合并成一个超级集群？我们需要实验。

#### 尚未解决的问题

- **同质性偏好的定义**：什么是“相似输出”？用向量相似度还是关键词匹配？这本身就会引入某种预设的相似性标准，而这个标准可能影响分化的方向。但也许这个标准可以从历史数据中自适应？
- **隔离的稳定性**：隔离后的子网络如何避免被偶然的跨界消息瓦解？是否需要一个“边界维护”机制，比如对外部消息的默认忽略？这又可能引入新的设计。
- **多层级分化**：子网络内部可能进一步分化，形成嵌套结构。这种层次性能否自然涌现？
- **与外部输入的互动**：inputoutputagent的消息来自现实世界，这些消息可能同时触及多个子网络，破坏隔离。这是否意味着我们需要一个更复杂的语境选择机制，来“分配”输入到合适的子网络？

这些问题没有现成答案。我们只能继续追问，并期待实验给出意想不到的启示。

## 核心组件详解

### 1. Agent：语义处理器的基本单元

`driver/agent.py` 里的 `Agent` 类是整个系统的原子。每个 Agent 本质上是一个被预提示词（pre-prompt）严格约束的 LLM 调用包装器。

**为什么这样设计？** 因为我们想最小化模型内部知识对计算的贡献。通过预提示词，我们把 LLM 从"无所不知的对话者"降级为"字符串转换器"——它接收特定格式的输入，输出特定格式的响应。

一个 Agent 的核心状态包括：

- **state**：一个字符串，Agent 自己维护的"记忆"。每轮处理时，上一轮的 state 会被注入到系统提示词中，Agent 需要在输出中更新它。这模拟了循环神经网络的隐状态传递。
- **input_connection**：谁可以给我发消息？格式是 `(sender_id, keyword)` 的列表。
- **output_connection**：我可以给谁发消息？格式是 `(keyword, receiver_id)` 的列表。
- **input_queue**：异步消息队列，其他 Agent 发来的消息在这里排队等待处理。

Agent 的处理循环（`_processing_loop`）是一个经典的"检查-处理-等待"模式：

```python
while running:
    try:
        message = await input_queue.get(timeout=interval)
        # 收集同一时间段内的所有消息
        messages = collect_batch_messages()
        # 调用 LLM 处理
        await process_messages_batch(messages)
    except TimeoutError:
        pass  # 没有消息，继续等待
```

注意这里的**批处理设计**——我们不希望 Agent 对每条消息都立即响应，而是积累一小段时间内的所有输入，一次性处理。这有两个好处：一是减少 LLM 调用次数，二是让 Agent 能看到同一"时刻"的多个输入。

### 2. MessageBus：网络的神经系统

`driver/agent_system.py` 里的 `MessageBus` 负责在 Agent 之间传递消息。它很简单，就是一个字典 `agents: Dict[str, Agent]` 加上一个异步发送方法。

但简单不代表不重要。MessageBus 是系统**去中心化**的关键——Agent 之间不直接通信，而是通过总线中转。这意味着：

1. Agent 不需要引用其他Agent。
2. 我们可以在总线层插入延迟、暂停、甚至丢弃消息，来模拟网络的不确定性。
3. 日志和监控可以在这一层统一处理。

MessageBus 还提供了全局控制功能：`pause_messages()` 和 `set_message_delay()`。这让你可以像慢放录像一样观察系统的行为，对于调试和理解网络动态非常有用。

### 3. AgentSystem：网络的管理者

`AgentSystem` 是 MessageBus 的包装者，也是所有 Agent 的容器。它负责：

- 注册和注销 Agent
- 启动和停止所有 Agent 的处理循环
- 管理 I/O Agent（与外界交互的特殊 Agent）

这里有一个设计决策值得注意：**AgentSystem 不干预 Agent 的具体行为**。它只负责"启动"和"停止"，不告诉 Agent"该做什么"。

### 4. AgentNetwork：连接拓扑的构建者

`driver/net.py` 里的 `AgentNetwork` 负责创建 Agent 之间的连接关系。目前它使用随机初始化：每个 Agent 随机连接到 1-3 个其他 Agent，关键词也是随机生成的。

这看起来有点简陋，但是有意为之（不过之后也会使用其他随机初始化方法，例如无尺度网络）。因为我们希望**连接关系是演化的，而不是预设的**。随机初始化提供了一个"原始汤"，之后的连接调整（通过信号系统）才是网络结构涌现的关键。

AgentNetwork 还负责把 I/O Agent 接入网络。输入 Agent（如 TerminalInputAgent）会被随机连接到一个内部 Agent，输出 Agent 同理。这些连接被标记为"受保护"，防止被信号系统切断——毕竟，如果系统失去了与外界的连接，它就变成了纯粹的自说自话。

### 5. I/O Agent：系统与外界的边界

`driver/i_o_agent.py` 定义了两个抽象基类：`InputAgent` 和 `OutputAgent`。

- **InputAgent**：从外部收集数据，发送到网络。它必须实现 `has_data_to_send()` 和 `collect_data()`。
- **OutputAgent**：从网络接收数据，输出到外部。它必须实现 `explore()` 和 `execute_data()`。

这种抽象让系统可以与任何外部接口对接。目前的实现包括：

- **TerminalInputAgent/TerminalOutputAgent**：把终端的渲染输出作为输入，把 Agent 的输出作为终端命令。
- **UserInputAgent/ConsoleOutputAgent**：简单的控制台输入输出，用于测试。

I/O Agent 是系统**具身性**的来源。没有它们，Agent 网络只是在自我循环；有了它们，Agent 可以"看到"和"影响"外部世界。

## 终端子系统：AVBash 的设计

`AVBash/` 目录包含一个完整的多窗口终端模拟器。这是服务于一个核心目标：**给 Agent 提供类似人类的 shell 使用体验**。

### 为什么需要终端模拟？

我们希望 Agent 能操作计算机，但直接让 Agent 执行系统命令意味着Agent必须通过自己的知识理解一切。通过终端模拟器，我们可以：

1. **隔离**：每个窗口运行独立的 bash 进程，Agent 的"操作"只是向终端发送字符，终端再转发给 shell。
2. **可观察**：Agent 可以看到终端的完整渲染输出（包括光标位置、颜色、布局），而不只是 stdout，通过人性化处理终端渲染，即可帮助系统进行学习。
3. **可控**：我们可以暂停、回放、甚至修改终端的行为，来研究 Agent 的学习过程。

### 架构概览

```
AVBash/
├── terminal.py          # 核心：Window 和 TerminalManager 类
├── terminal_agents.py   # 包装：TerminalInputAgent 和 TerminalOutputAgent
└── __main__.py          # 独立运行入口
```

**Window 类**：管理单个 bash 会话。它使用 Python 的 `pty` 模块创建伪终端，然后启动 bash 进程。所有输入输出都通过文件描述符进行，Window 负责维护屏幕缓冲区、处理滚动、搜索历史等功能。

**TerminalManager 类**：管理多个 Window，处理用户输入（包括控制命令如 `/new`、`/focus`），并定时渲染所有窗口的状态。渲染输出是一个纯文本字符串，包含了所有窗口的当前状态——这个字符串会被 TerminalInputAgent 捕获并发送到 Agent 网络。

**控制命令**：用户（或 Agent）可以通过 `/` 开头的命令控制终端。比如 `/new 项目A` 创建一个新窗口，`/focus 2` 切换到窗口 2，`/list` 查看所有窗口状态。这些命令不是 shell 命令，而是由 TerminalManager 直接处理的。

## 监控与可视化：理解网络的运行

`utils/visual_monitor/` 目录包含一个基于 WebSocket 的实时监控系统。这在开发阶段至关重要——当你有几十个 Agent 在互相发消息时，你需要一种方式"看到"网络里发生了什么。

### 日志系统的设计哲学

我们使用 JSONL 格式记录所有事件，每条日志是一个 JSON 对象，包含时间戳、事件类型、数据来源和详细信息。这种格式便于机器解析，也便于人类阅读（用 `jq` 工具可以很方便地过滤和查询）。

日志分为三个级别：

- **CONTENT**：只记录关键事件（Agent 创建、消息发送、连接变更）。
- **DETAIL**：记录更多细节（消息内容摘要、状态更新）。
- **ARCH**：记录架构级信息（异步任务创建、对象引用地址），用于深度调试。
*该系统尚不确定是否完整实现*

`UnifiedLogger` 是一个单例类，所有组件都通过它记录日志。`Loggable` 基类提供了便捷的包装方法，让子类可以用 `self.info()`、`self.debug()` 等方式记录。

### Web 监控界面

`server.py` 启动一个 WebSocket 服务器，同时提供 HTTP 静态文件服务。浏览器连接到 WebSocket 后，可以实时收到日志更新，并渲染成可视化的网络拓扑图。

监控界面显示：

- **网络拓扑**：Agent 之间的连接关系，用有向图表示。
- **消息流**：实时的消息传递动画，让你看到信息在网络中的流动。
- **Agent 状态**：每个 Agent 的当前状态、激活频率、输入输出连接。
- **异步状态**：当前运行的异步任务，用于调试并发问题。

## 信号系统：网络的自组织能力

这是整个系统中最微妙也最重要的部分。Agent 不仅可以发送"内容消息"，还可以发送"信号"来调整网络结构。

### 信号类型

目前支持四种信号，都在 `Agent.process_signal()` 中处理：

1. **REJECT_INPUT**：断开某个输入连接。用于"我不需要这个信息源了"。
2. **ACCEPT_INPUT**：修改输入连接的关键词。用于"换个方式理解这个输入"。
3. **SET_OUTPUT**：修改输出连接的关键词。用于"换个方式标记我的输出"。
4. **REJECT_OUTPUT**：断开某个输出连接。用于"我不需要发给这个接收者了"。

### 为什么信号是 JSON 格式？

信号需要被系统解析并执行特定的操作，所以必须结构化。我们选择了 JSON，因为它简单、通用、易于扩展。信号被包裹在 `<signal>...</signal>` 标签中，Agent 的响应被解析后，匹配到的信号会被提取出来处理。

### 权限控制

I/O Agent 的连接是受保护的——普通 Agent 不能通过信号切断它们。这是为了防止 Agent"自我封闭"，失去与外界的连接。在代码中，这通过 `can_delete_connection()` 方法实现：如果目标 Agent 是 InputAgent 或 OutputAgent，会调用它的 `on_connection_delete_request()` 方法，而默认实现是返回 `False`（拒绝删除）。

## 预提示词：约束的艺术

`driver/pre_prompt.md` 是系统的"宪法"。它定义了 Agent 应该如何理解自己的角色、如何处理输入、如何更新状态、如何使用信号。

### 核心约束

1. **你不是助手**：明确告诉 Agent 它不是来聊天的，而是网络中的一个处理节点。
2. **状态必须更新**：每轮输出必须包含 `<self_state>`，否则 Agent 会"失忆"。
3. **关键词必须在列表中**：只有 `output_connection` 中存在的 keyword 才会被系统识别。
4. **记忆要详细**：鼓励 Agent 像写日记一样记录具体细节，而不是抽象概括。

### 为什么这样设计？

预提示词是系统"语义约束"的具体实现。我们希望通过严格的提示词，把 LLM 从"生成内容的对话者"变成"处理字符串的函数"。这不是要限制 LLM 的能力，而是要**明确它的职责边界**——在这个系统中，Agent 的责任是维护状态、传递信息、调整连接，而不是"理解"或"思考"。

当然，LLM 还是会"思考"的，但这种思考被引导向特定方向：关注信息流的效率，而不是内容的"正确性"；关注连接的质量，而不是输出的"质量"。

## 启动流程：从代码到运行中的系统

`main.py` 是应用的入口。启动流程如下：

1. **归档旧日志**：防止日志文件无限增长。
2. **启动监控服务器**：在后台运行 WebSocket 服务器。
3. **创建终端**：初始化 TerminalManager，创建第一个默认窗口。
4. **创建 AgentSystem**：这是整个网络的管理者。
5. **创建 AgentNetwork**：初始化内部 Agent（默认 5 个）和它们的随机连接。
6. **创建 I/O Agent**：从终端获取 InputAgent 和 OutputAgent。
7. **连接 I/O Agent**：把终端 Agent 接入网络。
8. **启动所有 Agent**：开始处理循环。

启动后，系统进入运行状态。你可以访问 `http://localhost:8765` 查看监控界面。

## 设计决策的回顾

在实现过程中，我们做了一些关键选择：

### 为什么用 asyncio？

Python 的 asyncio 提供了轻量级的并发，让我们可以同时运行几十个 Agent 的处理循环，而不需要为每个 Agent 开一个线程。Agent 的处理循环大多是 IO 密集型的（等待 LLM 响应），所以 asyncio 的协程模型非常适合。


### 为什么状态是字符串？

Agent 的 state 是一个简单的字符串，而不是结构化数据（如 JSON）。这是有意为之——我们希望 Agent 自己决定如何编码和解码它的记忆，而不是我们替它规定格式。这给了 Agent 更大的灵活性，也带来了一定的风险（Agent 可能生成格式混乱的状态），但我们认为这是"涌现"的必要代价。

### 为什么关键词是字符串？

Agent 之间的连接通过"关键词"来标识。关键词是字符串，Agent 可以用任何它喜欢的词作为关键词。这模拟了自然语言中"标签"的概念——Agent 可以动态地创建、修改、废弃标签，而不需要中央协调。


# 当前实验情况
目前的实验创建了一个5个节点的简单网络，随机添加边，加入一个输入Agent，一个输出Agent，连接到终端，在提示词中提醒了Agent发送指令（否则，连接到输出的那个Agent会不会输出就只能看运气了）
Agent展示了基本的认知能力，但是很不幸它们总是觉得需要“打招呼”“握手”“询问”，然而这些都是不可能的。
在多次实验中，Agent出现的行为多样性不算平凡，但是也明显无法达到human-like的水平。在之前的实验中，提示词没有提醒Agent发送终端指令，然而一个连接到输出的Agent通过输入的模式，猜到了自己需要输出ls这样的指令。但是这种情况实际上比较稀有。
如果仔细看log的话，可以发现Agent的行为的确会受到网络拓扑结构的影响。验证了我们的基础假设。但是5个节点实在太少，网络非常容易陷入停滞状态。
接下来我会扩大实验规模，我预想此时Agent的激活可能出现爆炸（因为提示词里要求了它进行探索性输出），因此我会看情况加入一些缓解措施。
扩大规模后，我们需要关注网络初始化的问题，我可能会换成无标度网络这样的模型。预想的目标是验证深度信息传递是否会产生更多的多样性，否则模型会收敛到使用Bash却没有目标。
扩大规模后，系统的内部状态会更丰富，“想象”的空间更大，因此更有可能产生探索性输出——我认为探索不该有目的地的限制。但是我也需要加入更多系统限制，例如修剪那些提前进入老生代的Agent
现在的问题仍然是LLM模型太过于受限，输出的模式很固定。（唉，即使是对话我都会觉得LLM太单调了，明明是学了全人类全互联网的语料——然而基座模型却很难做指令遵从。这让我有一种得做提示词优化（无论是不是梯度优化）的想法）
此外，从log里可以看出LLM的状态收敛速度比较快，这不是很好，未来可能需要干预，将固定下来的状态进行修改、变异，做为新神经元的状态。
